## 🎯 Общая концепция

### Что такое роли?

**Роль** - это группа прав доступа, которая определяет, что может делать пользователь в системе.

**Примеры ролей:**
- `admin` - полный доступ ко всему
- `editor` - может редактировать контент
- `user` - базовый доступ

### Что такое пользователь?

**Пользователь** - это человек, который может войти в админку. У каждого пользователя:
- Имя (name)
- Email (логин)
- Пароль
- Роль (role_id)

---

## 🗄️ Структура базы данных

### Таблица `roles`

Хранит все роли в системе.

**Структура:**
```sql
CREATE TABLE roles (
    id BIGINT PRIMARY KEY,
    name VARCHAR(255) UNIQUE,      -- 'Admin', 'Editor', 'User'
    slug VARCHAR(255) UNIQUE,       -- 'admin', 'editor', 'user'
    description TEXT,               -- Описание роли
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

**Пример данных:**
```
id | name   | slug   | description
---|--------|--------|------------
1  | Admin  | admin  | Полный доступ
2  | Editor | editor | Редактирование контента
```

---

### Таблица `users`

Хранит всех пользователей системы.

**Структура:**
```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name VARCHAR(255),             -- Имя пользователя
    email VARCHAR(255) UNIQUE,      -- Email (логин)
    password VARCHAR(255),          -- Хешированный пароль
    role VARCHAR(255),              -- Старая система (для обратной совместимости)
    role_id BIGINT,                 -- Связь с roles.id
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    FOREIGN KEY (role_id) REFERENCES roles(id)
);
```

**Пример данных:**
```
id | name      | email              | password (hash) | role_id
---|-----------|--------------------|-----------------|--------
1  | Admin     | admin@example.com  | $2y$12$...     | 1
2  | Editor    | editor@example.com | $2y$12$...     | 2
```

**Связь:**
- `users.role_id` → `roles.id`
- Один пользователь = одна роль
- Одна роль = много пользователей

## 📦 Модели

### Модель `Role` (`app/Models/Role.php`)

**Отношения:**
```php
public function users()
{
    return $this->hasMany(User::class);
}
```

**Использование:**
```php
$role = Role::find(1);
$users = $role->users; // Все пользователи с этой ролью
```

---

### Модель `User` (`app/Models/User.php`)

**Отношения:**
```php
public function roleRelation()
{
    return $this->belongsTo(Role::class, 'role_id');
}
```

**Методы:**

#### 1. `hasRole($roleSlug)`
Проверяет, имеет ли пользователь указанную роль.

```php
if ($user->hasRole('admin')) {
    // Пользователь - админ
}
```

**Логика:**
1. Проверяет `role_id` → загружает `roleRelation` → сравнивает `slug`
2. Если нет `role_id` → проверяет колонку `role` (старая система)
3. Возвращает `true` или `false`

#### 2. `isAdmin()`
Проверяет, является ли пользователь администратором.

```php
if ($user->isAdmin()) {
    // Пользователь - админ
}
```

**Эквивалентно:** `$user->hasRole('admin')`

## 🔐 Логика работы

### 1. Создание роли

**Шаги:**
1. Заходите в `/admin/roles`
2. Нажимаете "Создать роль"
3. Заполняете:
   - **Название:** `Admin` (отображается в интерфейсе)
   - **Slug:** `admin` (используется в коде, только строчные буквы, цифры, дефис)
   - **Описание:** по желанию
4. Сохраняете

**Результат:**
- Создается запись в таблице `roles`
- Slug должен быть уникальным

---

### 2. Создание пользователя

**Шаги:**
1. Заходите в `/admin/users`
2. Нажимаете "Создать пользователя"
3. Заполняете:
   - **Имя:** `Admin User`
   - **Email:** `admin@example.com` (будет логином)
   - **Пароль:** `password123` (минимум 6 символов)
   - **Подтверждение пароля:** `password123`
   - **Роль:** выбираете из списка (например, "Admin")
4. Сохраняете

**Что происходит:**
1. Пароль хешируется: `Hash::make($password)`
2. Создается запись в таблице `users`
3. Устанавливается `role_id` = ID выбранной роли

**Результат:**
- Пользователь может войти по email или имени
- Пароль хранится в хешированном виде
- Пользователь имеет назначенную роль

---

### 3. Вход в админку

**Процесс:**

```
1. Пользователь вводит:
   - Логин: email ИЛИ имя
   - Пароль: пароль

2. AdminLoginController@login:
   ├─ Определяет тип: email или name?
   ├─ Ищет пользователя: User::where($field, $login)
   ├─ Проверяет пароль: Hash::check($password, $user->password)
   │
   ├─ Если пароль верный:
   │  ├─ Авторизует: Auth::login($user)
   │  ├─ Проверяет роль: $user->hasRole('admin')
   │  │
   │  ├─ Если есть роль → Редирект на /admin
   │  └─ Если нет роли → Ошибка "У пользователя не назначена роль"
   │
   └─ Если пароль неверный:
      └─ Ошибка "Неверный логин или пароль"
```

**Код:**
```php
// Определяем тип входа
$field = filter_var($login, FILTER_VALIDATE_EMAIL) ? 'email' : 'name';

// Ищем пользователя
$user = User::where($field, $login)->first();

// Проверяем пароль
if ($user && Hash::check($password, $user->password)) {
    Auth::login($user);
    
    // Проверяем роль
    if ($user->role_id || $user->attributes['role']) {
        return redirect('/admin');
    }
}
```

---

### 4. Проверка доступа

**Middleware `auth`:**
- Проверяет: залогинен ли пользователь?
- Если нет → редирект на `/login`
- Если да → пропускает дальше

**Middleware `admin`:**
- Проверяет: имеет ли пользователь роль `admin`?
- Если нет → вылогинивает и редирект на `/login`
- Если да → пропускает дальше

**Использование в роутах:**
```php
Route::middleware(['auth', 'admin'])->group(function () {
    Route::resource('roles', RoleController::class);
    Route::resource('users', UserController::class);
});
```

## 🎛️ Управление в админке

### Роли (`/admin/roles`)

**Доступ:** Только для пользователей с ролью `admin`

**Возможности:**
- ✅ Просмотр списка ролей
- ✅ Создание новой роли
- ✅ Редактирование роли
- ✅ Удаление роли (если нет пользователей с этой ролью)

**Важно:**
- Slug должен быть уникальным
- Slug может содержать только строчные буквы, цифры и дефис
- Нельзя удалить роль, если есть пользователи с этой ролью

---

### Пользователи (`/admin/users`)

**Доступ:** Только для пользователей с ролью `admin`

**Возможности:**
- ✅ Просмотр списка пользователей
- ✅ Создание нового пользователя
- ✅ Редактирование пользователя
- ✅ Изменение пароля
- ✅ Назначение/изменение роли
- ✅ Удаление пользователя (нельзя удалить самого себя)

**Особенности:**
- Пароль хранится в хешированном виде (нельзя показать)
- Можно изменить пароль при редактировании
- Можно назначить роль из списка существующих ролей

## 💡 Примеры использования

### В контроллере

```php
// Проверка роли
if (auth()->user()->hasRole('admin')) {
    // Только для админов
}

// Проверка, является ли админом
if (auth()->user()->isAdmin()) {
    // Только для админов
}

// Получить роль пользователя
$user = auth()->user();
$role = $user->roleRelation; // Объект Role
echo $role->name; // "Admin"
echo $role->slug; // "admin"
```

```blade

@endif

@if($user->roleRelation)
    <span class="badge">{{ $user->roleRelation->name }}</span>
@endif
```

---

### В middleware

```php
// Проверка в middleware
if (!$user->hasRole('admin')) {
    return redirect('/login');
}
```

## 🛡️ Безопасность

### Хеширование паролей

**Пароли НИКОГДА не хранятся в открытом виде!**

```php
// При создании пользователя
$user->password = Hash::make('password123');
// Сохраняется: $2y$12$nELmjL1yq.whZXr1OYjN10Cw0DXCcsdYxJ2UbxSEpXKLI./qbdvXC

// При проверке
Hash::check('password123', $user->password); // true
Hash::check('wrong', $user->password);      // false
```

**Важно:**
- Нельзя восстановить исходный пароль из хеша
- Один и тот же пароль может давать разный хеш (из-за соли)
- Используется алгоритм bcrypt

---

### Проверка доступа

**Двухуровневая защита:**

1. **Middleware `auth`:**
   - Проверяет сессию
   - Если нет сессии → редирект на `/login`

2. **Middleware `admin`:**
   - Проверяет роль
   - Если роль не `admin` → вылогинивает и редирект на `/login`

**Результат:**
- Только залогиненные пользователи могут видеть админку
- Только пользователи с ролью `admin` могут управлять ролями и пользователями

## 📝 Важные моменты

### 1. Slug роли

**Slug** - это идентификатор роли в коде.

**Правила:**
- Только строчные буквы: `admin`, `editor`
- Можно использовать цифры: `admin2`
- Можно использовать дефис: `super-admin`
- Должен быть уникальным

**Использование:**
```php
$user->hasRole('admin'); // Проверяет slug, не name!
```

---

### 2. Обратная совместимость

Система поддерживает две схемы:

**Новая система (рекомендуется):**
- `users.role_id` → `roles.id`
- Используется отношение `roleRelation()`

**Старая система (для обратной совместимости):**
- `users.role` → строка ('admin', 'user')
- Используется напрямую из колонки

**Метод `hasRole()` проверяет обе системы:**
```php
// Сначала проверяет role_id
if ($this->role_id && $this->roleRelation) {
    return $this->roleRelation->slug === $roleSlug;
}

// Потом проверяет колонку role
$roleValue = $this->attributes['role'] ?? null;
if ($roleValue && is_string($roleValue)) {
    return $roleValue === $roleSlug;
}
```

---

### 3. Вход по email или имени

**Логика:**
```php
// Определяем тип
$field = filter_var($login, FILTER_VALIDATE_EMAIL) ? 'email' : 'name';

// Ищем пользователя
$user = User::where($field, $login)->first();
```

**Примеры:**
- `admin@example.com` → ищет по `email`
- `Admin User` → ищет по `name`

## 🎯 Итого

### Что нужно помнить:

1. **Роль** = группа прав доступа (хранится в `roles`)
2. **Пользователь** = человек с логином и паролем (хранится в `users`)
3. **Связь:** `users.role_id` → `roles.id`
4. **Проверка:** `$user->hasRole('admin')`
5. **Доступ:** только `admin` может управлять ролями и пользователями
6. **Безопасность:** пароли хешируются, доступ проверяется через middleware

### Стандартный процесс:

1. Создать роль (slug: `admin`)
2. Создать пользователя с этой ролью
3. Войти по email/имени и паролю
4. Управлять ролями и пользователями (если роль `admin`)
